# QuillToDocx

Converts Quill `.doc` files into Word `.docx` files. Run `quilltodocx <basename>` or `quilltodocx <input.doc> [output.docx]`.

QuillToDocx exists to make old Quill word-processor documents easier to inspect, preserve, and share on current systems. The original file format is obscure and not well supported by modern tools, so this utility extracts Quill text and layout data and emits a standard Office Open XML document that opens directly in Word and compatible editors. It is intentionally small and direct: it reads the binary Quill structures, decodes paragraphs and inline formatting, and writes a usable DOCX with sensible defaults. That makes it practical both as a conversion tool and as a reference implementation for understanding the Quill document structure.

Internally, the program is split into a small set of focused F# modules that mirror the conversion pipeline. `QuillTypes.fs` defines the in-memory model for both the raw Quill file structures and the decoded document representation. The raw side includes records for the file header, paragraph table entries, layout table, tab definitions, and the container `QuillFile`. The decoded side separates parsing from rendering by introducing `TextStyle`, `Inline`, `DecodedParagraph`, and `DecodedDocument`. That distinction is important: the parser reads bytes and offsets exactly as stored on disk, while the later stages work with text runs, tabs, and style flags without needing to know anything about binary offsets.

`QuillBinary.fs` is the lowest-level reader. It handles big-endian integer extraction and maps fixed binary layouts onto F# records. Quill stores multi-byte values in big-endian order, so the helpers here read `uint16` and `uint32` values safely from a byte array. This module also contains the record constructors for header, paragraph, and layout records. The rest of the program relies on it to avoid repeating offset math throughout the codebase. By keeping those reads centralized, the parser stays much easier to audit against the source format.

`QuillParser.fs` sits on top of the binary reader and is responsible for turning an on-disk file into a `QuillFile`. It validates that the file is long enough to contain a header, checks the Quill identifier, slices out the text buffer, interprets the paragraph table, and reads the layout area. It also walks the Quill tab-table region after the layout record and converts it into a list of tab definitions. This module is deliberately conservative: it performs bounds checks, validates table sizes, and fails fast if the file shape looks wrong. That makes malformed or partially understood files fail predictably instead of producing silent corruption downstream.

`QuillDecode.fs` transforms raw Quill content into readable paragraphs. It knows about the Quill inline control bytes such as end-of-text, end-of-paragraph, tab, bold, underline, subscript, superscript, and soft hyphen. As it scans each paragraph byte stream, it maintains a current style state and flushes accumulated text into `RunText` items whenever a style toggle or control byte appears. The character decoding table also lives here. For normal ASCII bytes, decoding is direct; for Quill’s extended byte range, the program now maps bytes explicitly to Unicode code points rather than relying on source-file encoding artifacts. This avoids the mojibake that had previously shown up in converted documents.

The decoder also contains fallback and cleanup logic. If the paragraph table does not produce usable content, it can reconstruct paragraphs directly from the text buffer by splitting on Quill paragraph terminators. That makes the tool more tolerant of imperfect source metadata. At the end of the decoded body, it trims marker paragraphs that match the observed Quill trailer pattern, which prevents junk like `-0ÿÿÿÿ` from being emitted as a final paragraph in the DOCX. This is done at the decode stage rather than in the writer so that later stages operate on a clean logical document.

`Program.fs` provides the command-line interface. The CLI is intentionally simple, but it has a small amount of path-resolution logic to reduce friction. If the user runs `quilltodocx libcport`, the program resolves that as `libcport.doc` input and `libcport.docx` output. If the user passes `quilltodocx file.doc`, it writes `file.docx` by default. A fully explicit `input output` mode is also supported. After argument resolution and a file-exists check, `Program.fs` simply invokes the parser, decoder, and writer in sequence and reports success or a single top-level error.

`DocxWriter.fs` is the rendering stage. It uses the Open XML SDK to create a Wordprocessing document and populate its main document part. The writer converts Quill justification values into Word paragraph justification, turns Quill tab tables into `w:tabs`, and emits runs for plain text, tab characters, and soft hyphens. Inline style flags become run properties such as bold, underline, subscript, and superscript. A small styles part is also generated so the output has stable defaults instead of depending entirely on Word’s implicit formatting behavior.

Recent changes made the DOCX output more faithful and more predictable. The writer now sets explicit document defaults for font and paragraph spacing, applies a `Normal` paragraph style to body and header/footer paragraphs, and derives line spacing and paragraph spacing from Quill’s `LineGap` when that value appears reasonable. It also emits page size, margins, and page-number start information in the section properties, while clamping questionable source values to sane ranges. This balance is deliberate: the program tries to preserve Quill intent where the metadata is credible, but it avoids copying obviously implausible values directly into the DOCX when doing so would damage the layout.

Taken together, the design is straightforward: parse the binary file into structured data, decode that data into document semantics, and render those semantics to DOCX. Each step is narrow in scope, which keeps the program understandable and makes format experiments easier. If Quill support is extended further, the natural places to do that are already clear: binary-structure work belongs in `QuillBinary.fs` and `QuillParser.fs`, semantic cleanup belongs in `QuillDecode.fs`, and visual fidelity work belongs in `DocxWriter.fs`.
